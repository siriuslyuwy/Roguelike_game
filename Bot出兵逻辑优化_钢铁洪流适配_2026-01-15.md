# Bot 出兵逻辑优化 - 钢铁洪流适配

## 修改时间
2026-01-15 00:30

## 问题诊断

### 核心矛盾
**人类玩家**：钢铁洪流强得可怕（费用/CD减半，暴兵流碾压）  
**Bot 测试**：钢铁洪流通关率 0%，平均层数仅 4.67

### 根本原因
Bot 的出兵逻辑是为"标准游戏"设计的，但钢铁洪流**改变了游戏规则**：
- 单位数量翻倍
- 单位质量下降（HP/伤害-25%）
- 核心玩法：**用数量优势压制**

Bot 的 4 个致命缺陷在钢铁洪流下被无限放大：

---

## 问题详解

### 问题 1：硬编码的坦克判定

**旧逻辑**（第 715 行）：
```python
def is_tank(k: str) -> bool:
    return ... and ut.hp >= 200  # 硬编码阈值
```

**问题**：
- 战士：160 × 0.75 = **120 HP** → 不是坦克
- 大锤：250 × 0.75 = **187.5 HP** → 不是坦克
- 狂战：220 × 0.75 = **165 HP** → 不是坦克
- Bot 认为"没有合格的坦克"，陷入恐慌

**人类玩法**：
- 3 个 120HP 的战士 = 360 HP 的血量池
- 数量就是质量

---

### 问题 2："存活计数" vs "产出速率"

**旧逻辑**（第 691-743 行）：
```python
tank_count = sum(1 for u in left_units if is_tank_unit)
if tank_count < max(1, back_count):
    # 补坦克
```

**问题**：
- Bot 关注"当前场上还有多少坦克"
- 钢铁洪流的坦克 HP-25%，死得很快
- `tank_count` 瞬间归零 → Bot 再补一个 → 又死了
- 陷入"单兵送人头"循环

**人类玩法**：
- 关注"压制力"而非"存活数"
- 源源不断补充，形成集群效果

---

### 问题 3："进攻等待"机制的负优化

**旧逻辑**（第 788-798 行）：
```python
if not game.can_spawn(..., pick):
    self._pending_spawn_key = pick
    return  # 等待攒钱，下次再出
```

**问题**：
- Bot 选中一个"理想单位"（如盾卫），钱不够就等待
- 钢铁洪流的核心优势是 **CD 减半 = 高频产出**
- 等待机制强行把高频变成了低频，浪费红利

**人类玩法**：
- 哪个亮了点哪个
- 疯狂点击，把 2 倍产量转化成 2 倍压力

---

### 问题 4：密度判定失效

**旧逻辑**（第 707 行）：
```python
density = len(right_units)  # 敌方数量
if density >= 4:
    # 出 AOE
```

**问题**：
- 钢铁洪流下，双方单位数都是正常的 2-3 倍
- `density` 一直处于"报警"状态
- Bot 频繁触发紧急逻辑（保守且低效）

---

## 解决方案

### 修改 1：动态坦克判定（治本）

**新逻辑**：
```python
def is_tank(k: str) -> bool:
    ut = UNIT_TYPES.get(k)
    if not ut:
        return False
    
    # 排除远程/治疗/增益
    if ut.is_ranged or ut.is_healer or getattr(ut, "is_buffer", False):
        return False
    if float(getattr(ut, "damage", 0)) <= 0:
        return False
    
    # 优先：基于 tags 判定（策划标记的就是坦克）
    if "tank" in ut.tags:
        return True
    
    # 兼容：动态 HP 阈值
    hp_threshold = 100 if is_steel_tide else 200
    return ut.hp >= hp_threshold
```

**效果**：
- 战士/大锤/狂战在钢铁洪流下仍被识别为坦克
- 可扩展：未来任何"改变HP"的祝福都能适配

---

### 修改 2：钢铁洪流专属逻辑

**新逻辑**：
```python
# 强制进入"高频暴兵"模式
is_steel_tide = (blessing == "steel_tide")
if is_steel_tide:
    is_urgent = True  # 不等待攒钱，立即出能出的

# 出兵优先级
if is_steel_tide:
    # 哪个能出就出哪个，轻微倾向近战（60%）
    melee_keys = [k for k in keys if is_tank(k)]
    if melee_keys and self.rng.random() < 0.6:
        pick = melee_keys[0]
    else:
        pick = keys[0]
```

**效果**：
- 禁用"等待攒钱"机制
- 最大化利用 CD 减半的优势
- 模拟人类"疯狂点击"的玩法

---

### 修改 3：动态前排比例

**新逻辑**：
```python
# 钢铁洪流：前排要求更宽松
min_tank_ratio = 0.5 if is_steel_tide else 1.0

if tank_count < max(1, int(back_count * min_tank_ratio)):
    # 补前排
```

**效果**：
- 标准模式：前排 ≥ 后排（1:1）
- 钢铁洪流：前排 ≥ 后排 × 0.5（1:2 即可）
- 允许更激进的后排配置

---

### 修改 4：提高密度阈值

**新逻辑**：
```python
density_threshold = 6 if is_steel_tide else 4
if density >= density_threshold:
    # 出 AOE
```

**效果**：
- 钢铁洪流下，敌人 6+ 才触发 AOE
- 避免过度保守

## 设计理念

### 核心思路
**祝福适配器模式**：为"规则改变型祝福"添加专属逻辑

### 可扩展性
未来添加新祝福时，只需：
1. 在 `is_steel_tide` 附近添加新的祝福检测
2. 在出兵逻辑中添加对应的适配分支
3. 在 Bot 类注释中记录适配说明

### 适配优先级
- **T0（必须适配）**：改变单位属性的祝福（钢铁洪流、精兵简政）
- **T1（建议适配）**：改变出兵成本的祝福（破釜沉舟）
- **T2（可选适配）**：改变战术玩法的祝福（教官光环）

## 其他祝福的适配建议

### 精兵简政 (elite_simplicity)
**机制**：限制 3 兵种，全属性+35%  
**问题**：Bot 可能解锁过多兵种，触发惩罚  
**建议**：添加"兵种数量上限检测"，优先升级而非解锁

### 掠夺者逻辑 (looter_logic)
**机制**：击杀获金  
**问题**：前期收益太少，Bot 可能因缺钱崩盘  
**建议**：添加"开局金币补偿"或"更激进的出兵策略"

### 战术大师 (tactical_master)
**机制**：技能 CD-25%  
**问题**：收益太迟，前期无法体现  
**建议**：添加"开局送 1 个技能"或"技能 CD-40%"

## 总结

### 关键洞察
**Bot 测试的价值 = Bot 逻辑与祝福玩法的适配度**

如果 Bot 不理解祝福的核心玩法，测试结果就是误导性的。

### 成功标准
- 每个祝福的通关率 > 0%
- 强势祝福（人类体验好）在 Bot 测试中也应表现优秀
- 弱势祝福（人类体验差）在 Bot 测试中也应表现一般

## 附录：坦克判定的演进

### V1（旧版）：硬编码 HP
```python
return ut.hp >= 200
```
**问题**：无法适配任何改变 HP 的祝福

### V2（当前）：Tags + 动态阈值
```python
if "tank" in ut.tags:
    return True
hp_threshold = 100 if is_steel_tide else 200
return ut.hp >= hp_threshold
```
**优势**：
- 策划通过 tags 定义坦克，Bot 自动识别
- 动态阈值适配不同祝福
- 可扩展性强

### V3（未来）：基于相对比例
```python
# 坦克 = HP 是队伍平均 HP 的 1.5 倍以上
avg_hp = sum(ut.hp for ut in my_units) / len(my_units)
return ut.hp >= avg_hp * 1.5
```
**优势**：完全动态，适配任何祝福组合
