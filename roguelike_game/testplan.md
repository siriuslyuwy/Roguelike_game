# 测试方案（整局 Run Bot）- v1

本文件描述**在持续改数值/内容的过程中**，如何用“整局 Run Bot”实现**可复现、可批量、可对比**的测试流程，用于平衡：
- 兵种数值
- Combo 卡
- 商店/事件
- 技能
- UI/音乐（主要做冒烟与回归，非核心平衡）

> 核心原则：**Build 随机，但执行不随机**。  
> 也就是：玩家侧 build 选择“均匀随机覆盖”，但局内出兵/基础运营用“中等水平执行”，避免把“乱玩导致输”误判为“build 弱/难度低”。

---

## 1. 测试目标与输出

### 1.1 平衡曲线报告（主报告）
- **通关率**：20 层通关率、平均到达层数、P50/P90 到达层数
- **经济曲线**：每层金币均值、战斗金币、事件净收益、商店净支出、刷新次数分布
- **成长曲线**：每层平均兵种数量、平均等级、技能数量、Combo 张数
- **压力曲线**：基地损失（总 HP 或被破线数）、战斗耗时

### 1.2 策略空间报告（验证“微操价值”）
对同一批 seed / 版本，比较不同策略档位 Bot：
- **弱 Bot（S0）**：有基础理性但缺乏编队/克制（下限）
- **中 Bot（S1/S2）**：本方案的**平衡基准**（接近“会玩的普通玩家”）
- **强 Bot（S3）**：在中 Bot 基础上加入更强克制与时机（上限）

关注差值：
- 若 **中/强 Bot 显著优于弱 Bot**：说明“编队/克制/时机”确实能赢数值，玩法成立。
- 若差值不明显：说明需要增强克制力度、单位定位或技能窗口收益。

---

## 2. Bot 总体结构（整局 Run）

整局 Run Bot 分为 4 个决策子系统：
- **地图走点（节点选择）**
- **事件 A/B（选项选择）**
- **商店（购买/刷新/捐赠/抢劫等）**
- **战斗（出兵战线 + 出兵序列 + 技能释放）**

本方案选择：
- **基准 Bot：中 Bot**
- **BuildPlan：均匀随机（玩家侧）**
- **敌人 AI：保持“低智商”，主要靠数值曲线驱动（不做复杂决策）**

---

## 3. BuildPlan（均匀随机，玩家侧）

### 3.1 BuildPlan 目标
每局开局从若干 BuildPlan 中**等概率**抽取一个 `build_plan_id`，并贯穿该 run：
- 节点倾向（走 shop/event/elite/rest 的偏好）
- 商店购买优先级
- Combo 候选选择倾向（按 tags）
- 事件选项倾向（在不自杀的前提下偏向推进 build）

### 3.2 建议的 10 个 BuildPlan（v1）
1) 标准编队（稳健、少刷新）  
2) AOE 清线（偏 AOE/聚怪收益）  
3) 速攻推进（偏速度/冲锋/侧翼）  
4) 经济滚雪球（偏金币/折扣/收益倍率）  
5) 技能成型（优先买技能/扩技能槽/拿重抽券）  
6) 锻造养成（优先锻造器与改锻收益）  
7) 俘虏成长（围绕俘虏收益/归顺升级）  
8) 防守反打（偏护盾/保命技能/稳健路线）  
9) 精英追收益（更倾向走精英，但仍保持中 Bot 操作）  
10) 随机但有体系（仅在“标签匹配集合”内随机，作为覆盖增强）  

> 注意：BuildPlan 影响“倾向”，不影响“基础执行能力”。战斗中依然按中 Bot 的编队/补线规则执行。

## 4. 中 Bot 执行规格（不靠“超聪明”，但不犯蠢）

### 4.1 战斗：最低执行能力（体现游戏精髓）
- **补崩线优先**：哪条线基地压力最大（前线最接近己方基地/基地掉血快），优先在该线出兵。
- **前排后排编队**：
  - 若该线前排不足：优先下肉盾/高 HP 近战。
  - 前排满足后：补后排输出（弓手/远程）。
  - 维持一个目标比例（示例：前排:后排 ≈ 2:1）。
- **AOE 反聚怪**：若同线敌人密度高/短腿多（更易聚集），提升 AOE 单位或 AOE 技能权重。
- **技能释放**：按“救急/窗口”触发（不做完美计算）：
  - 基地危险 → 防守技能优先（无敌/护盾）
  - 聚怪 → 黑洞/轰炸等控制或 AOE
  - 窗口爆发 → 全局 buff

### 4.2 商店：最低执行能力
- **预算与上限**：每次进店最多刷新 N 次（建议 N=2）。
- **购买优先级（可被 BuildPlan 调权重）**：
  - 补齐/升级核心兵种
  - 关键技能（未满槽时）
  - 锻造器/成长链条（若 BuildPlan 需要）
- **护栏**：若连续两次进店没有任何战力提升，则下一次强制购买“通用提升”（比如升级核心兵种）。

### 4.3 事件：最低执行能力
- 默认偏向“推进 BuildPlan”的选项，但不允许让金币跌到无法维持基本成长的危险区间（防自杀）。

### 4.4 地图走点：最低执行能力
- 使用“打分 + 少量随机扰动”：
  - shop/event/elite/rest/combat 各自有基础分
  - BuildPlan 对分数加成/减成
  - 基地危险时 rest 加大权重

## 5. 运行方式（建议实现为命令行参数）

### 5.1 建议参数
- `--seed <int>`：固定 seed（可复现）
- `--batch <N>`：跑 N 局整局 run（无渲染）
- `--bot <weak|mid|strong>`：选择策略档位
- `--build-uniform`：BuildPlan 均匀随机（玩家侧）
- `--out <path>`：输出 CSV/JSON

### 5.2 每局输出字段（最小集合）
- 版本号、seed、build_plan_id
- 是否通关、到达层数、总耗时
- 最终金币、商店刷新次数、商店净支出、事件净收益
- 最终技能数量、Combo 张数、兵种数量与平均等级
- 基地损失（总 HP 或被破线数）

---

## 6. 回归策略（持续改动时如何用）

### 6.1 小改动（单兵种/单卡/单价格）
- 中 Bot：100~200 局（build 均匀随机）
- 观察：通关率与经济曲线是否大幅偏移、是否出现“必拿/必死”模式

### 6.2 大改动（经济链、战斗结算、技能体系）
- 中 Bot：500~2000 局
- 弱/强 Bot：各 200~500 局（看策略空间差异）

### 6.3 黄金种子集（回归用）
- 维护 30~50 个固定 seed（含“最难/最离谱/最容易崩”的局）
- 每次合并大改前跑一遍：不崩溃、规则不违背、曲线不突变
